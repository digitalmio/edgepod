# CLI Type Extractor — Design Spec & Implementation Guide

## 1. Problem Statement

`@edgepod/client` currently exports untyped hooks:

```ts
const { data } = useQuery("getUsers"); // data: any
const { trigger } = useMutation("createUser"); // trigger: (...args: any[]) => any
```

When the frontend lives in a separate repository from the backend, it cannot import the backend's function types. TypeScript types exist only at compile time — they cannot be fetched over HTTP or read from a remote filesystem.

We need a CLI command that generates a **fully resolved, backend-agnostic type manifest** that the frontend can import.

## 2. Input Format

We scan **plain exported functions** from `edgepod/functions/index.ts`.

No wrapper function. The user writes plain TypeScript:

```ts
// edgepod/functions/index.ts
import { Ctx } from "../types";
import * as schema from "../schema";

export const getUsers = async (ctx: Ctx) => {
  return await ctx.db.select().from(schema.users);
};

export const getUserById = async (ctx: Ctx, args: { id: number }) => {
  const result = await ctx.db
    .select()
    .from(schema.users)
    .where(eq(schema.users.id, args.id))
    .limit(1);
  return result[0] || null;
};

export const createUser = async (ctx: Ctx, args: { email: string; name: string }) => {
  const newUser = await ctx.db.insert(schema.users).values(args).returning();
  return newUser[0];
};
```

**Why no wrapper?** A wrapper function (`edgepodFn({ ... })`) only makes export discovery slightly easier (~20 fewer lines of AST scanning). It does not solve the hard problem: resolving Drizzle's generic query-builder types into plain row shapes. We keep the DX minimal.

## 3. Output Format

### Generated File: `edgepod/client.gen.ts`

```ts
// Auto-generated by EdgePod CLI. Do not edit manually.

export type EdgePodRouter = {
  getUsers: {
    args: never;
    returns: Array<{ id: number; email: string; name: string | null }>;
  };
  getUserById: {
    args: { id: number };
    returns: { id: number; email: string; name: string | null } | null;
  };
  createUser: {
    args: { email: string; name: string };
    returns: { id: number; email: string; name: string | null };
  };
};
```

### Frontend Integration

The generated `EdgePodRouter` is passed to a typed client factory:

```ts
import type { EdgePodRouter } from "./edgepod/client.gen";
import { createEdgePodClient } from "@edgepod/client";

const { useQuery, useMutation } = createEdgePodClient<EdgePodRouter>();

// Fully typed — args and return type inferred from the generated router
const { data } = useQuery("getUsers");
//    ^? data: Array<{ id: number; email: string; name: string | null }>

const { trigger } = useMutation("createUser");
//    ^? trigger: (args: { email: string; name: string }) => Promise<...>
```

`createEdgePodClient` is a thin wrapper that returns `useQuery` and `useMutation` with the `functionName` parameter constrained to `keyof EdgePodRouter` and the return type mapped from the router.

## 4. TypeScript Compiler API Approach

We use the **built-in TypeScript compiler API** (`typescript`). No extra dependencies like `ts-morph`.

`ts-morph` is excellent for AST _manipulation_ (refactoring, adding imports). We only need _reading_ and _type resolution_. The built-in API is sufficient and avoids a ~2MB dependency in the CLI.

### 4.1 Program Construction

```ts
import ts from "typescript";
import path from "node:path";

export function createTypeExtractor(entryFile: string) {
  const configPath = ts.findConfigFile(path.dirname(entryFile), ts.sys.fileExists, "tsconfig.json");

  const { config, error } = configPath
    ? ts.readConfigFile(configPath, ts.sys.readFile)
    : { config: {}, error: undefined };

  if (error) {
    throw new Error(ts.flattenDiagnosticMessageText(error.messageText, "\n"));
  }

  const parsed = ts.parseJsonConfigFileContent(
    config,
    ts.sys,
    path.dirname(configPath || entryFile),
  );

  const program = ts.createProgram([entryFile], {
    ...parsed.options,
    noEmit: true,
    declaration: false,
  });

  return {
    checker: program.getTypeChecker(),
    sourceFile: program.getSourceFile(entryFile)!,
  };
}
```

**Critical:** We must use the user's `tsconfig.json`. Otherwise imports like `import * as schema from "../schema"` and `import { Ctx } from "../types"` will not resolve. Without resolution, `typeof schema.users.$inferSelect` will not expand into a concrete row shape.

### 4.2 Export Discovery

Handle both patterns:

- `export const foo = async (ctx, args) => ...`
- `export async function foo(ctx, args) { ... }`

```ts
function findExportedFunctions(sourceFile: ts.SourceFile, checker: ts.TypeChecker) {
  const functions: Array<{
    name: string;
    symbol: ts.Symbol;
    type: ts.Type;
  }> = [];

  ts.forEachChild(sourceFile, (node) => {
    if (!isNodeExported(node)) return;

    let symbol: ts.Symbol | undefined;

    // Pattern A: export const foo = async (ctx, args) => ...
    if (ts.isVariableStatement(node)) {
      const decl = node.declarationList.declarations[0];
      if (decl.initializer) {
        symbol = checker.getSymbolAtLocation(decl.name);
      }
    }

    // Pattern B: export async function foo(ctx, args) { ... }
    if (ts.isFunctionDeclaration(node) && node.name) {
      symbol = checker.getSymbolAtLocation(node.name);
    }

    if (symbol) {
      const type = checker.getTypeOfSymbolAtLocation(symbol, node);
      functions.push({ name: symbol.name, symbol, type });
    }
  });

  return functions;
}

function isNodeExported(node: ts.Node): boolean {
  return (
    (ts.getCombinedModifierFlags(node as ts.Declaration) & ts.ModifierFlags.Export) !== 0 ||
    (!!node.parent && node.parent.kind === ts.SyntaxKind.SourceFile)
  );
}
```

### 4.3 Signature Extraction

```ts
function extractFunctionRouterEntry(
  name: string,
  fnType: ts.Type,
  checker: ts.TypeChecker,
): { args: string; returns: string } {
  const callSignatures = fnType.getCallSignatures();

  if (callSignatures.length === 0) {
    throw new Error(`No call signatures found for function "${name}"`);
  }

  const sig = callSignatures[0];
  const params = sig.getParameters();

  // Skip ctx (params[0]), extract args (params[1])
  const argsParam = params[1];
  const argsType = argsParam
    ? checker.getTypeOfSymbolAtLocation(argsParam, argsParam.valueDeclaration!)
    : checker.getUndefinedType();

  // Unwrap Promise<T> from the return type
  let returnType = checker.getReturnTypeOfSignature(sig);
  returnType = unwrapPromise(returnType, checker);

  return {
    args: typeToString(argsType, checker),
    returns: typeToString(returnType, checker),
  };
}

function unwrapPromise(type: ts.Type, checker: ts.TypeChecker): ts.Type {
  if (type.symbol?.name === "Promise") {
    const typeRef = type as ts.TypeReference;
    if (typeRef.typeArguments && typeRef.typeArguments.length > 0) {
      return typeRef.typeArguments[0];
    }
  }
  return type;
}
```

## 5. The Hard Part: Recursive Type Inliner

### Why `checker.typeToString()` Is Insufficient

```ts
const returnType = checker.getReturnTypeOfSignature(sig);
checker.typeToString(returnType);
// → "typeof schema.users.$inferSelect"  ❌
// or "SQLiteSelect<...>"                ❌
// or "import("../schema").users"        ❌
```

None of these are valid in a generated file with zero backend imports. We need to emit:

```ts
{
  id: number;
  email: string;
  name: string | null;
}
```

### The Recursive Walker

```ts
function typeToString(
  type: ts.Type,
  checker: ts.TypeChecker,
  seen = new Set<number>(),
  depth = 0,
): string {
  if (depth > 10) return "any /* depth limit */";
  if (seen.has(type.id)) return "any /* circular */";
  seen.add(type.id);

  // Primitives
  if (type.flags & ts.TypeFlags.String) return "string";
  if (type.flags & ts.TypeFlags.Number) return "number";
  if (type.flags & ts.TypeFlags.Boolean) return "boolean";
  if (type.flags & ts.TypeFlags.Null) return "null";
  if (type.flags & ts.TypeFlags.Undefined) return "undefined";
  if (type.flags & ts.TypeFlags.Void) return "void";
  if (type.flags & ts.TypeFlags.Never) return "never";

  // Literal types
  if (type.isLiteral()) {
    return typeof type.value === "string" ? `"${type.value}"` : String(type.value);
  }

  // Arrays
  if (checker.isArrayType(type)) {
    const elementType = checker.getTypeArguments(type as ts.TypeReference)[0];
    return `Array<${typeToString(elementType, checker, new Set(seen), depth + 1)}>`;
  }

  // Tuples
  if (checker.isTupleType(type)) {
    const elements = checker
      .getTypeArguments(type as ts.TypeReference)
      .map((t) => typeToString(t, checker, new Set(seen), depth + 1));
    return `[${elements.join(", ")}]`;
  }

  // Unions
  if (type.isUnion()) {
    return type.types.map((t) => typeToString(t, checker, new Set(seen), depth + 1)).join(" | ");
  }

  // Intersections
  if (type.isIntersection()) {
    return type.types.map((t) => typeToString(t, checker, new Set(seen), depth + 1)).join(" & ");
  }

  // Objects (handles Drizzle $inferSelect, plain objects, mapped types)
  if (type.flags & ts.TypeFlags.Object) {
    const props = checker.getPropertiesOfType(type);
    const members = props.map((prop) => {
      const propType = checker.getTypeOfSymbolAtLocation(
        prop,
        prop.valueDeclaration || prop.declarations![0],
      );
      const optional = prop.flags & ts.SymbolFlags.Optional ? "?" : "";
      return `${prop.name}${optional}: ${typeToString(
        propType,
        checker,
        new Set(seen),
        depth + 1,
      )}`;
    });

    // Index signatures: { [key: string]: any }
    const stringIndex = checker.getIndexTypeOfType(type, ts.IndexKind.String);
    const numberIndex = checker.getIndexTypeOfType(type, ts.IndexKind.Number);
    if (stringIndex) {
      members.push(
        `[key: string]: ${typeToString(stringIndex, checker, new Set(seen), depth + 1)}`,
      );
    }
    if (numberIndex) {
      members.push(
        `[key: number]: ${typeToString(numberIndex, checker, new Set(seen), depth + 1)}`,
      );
    }

    return `{ ${members.join("; ")} }`;
  }

  // Fallback: try typeToString with flags that avoid import references
  return checker.typeToString(
    type,
    undefined,
    ts.TypeFormatFlags.NoTypeReduction | ts.TypeFormatFlags.UseFullyQualifiedType,
  );
}
```

## 6. Drizzle-Specific Edge Cases

Drizzle types are the primary source of complexity. The compiler API sees the **generic type parameters** of the query builder, not the resolved row shape. We must force resolution.

### 6.1 `$inferSelect` Resolution

When a function body contains:

```ts
return await ctx.db.select().from(schema.users);
```

The return type of the signature is `Promise<SQLiteSelect<...>>` (the query builder). After unwrapping `Promise`, we get `SQLiteSelect<...>`. This is a class instance type, not the row array.

**We need the resolved execution result, not the builder.**

#### Solution A: Special-case Drizzle builder types

If the type symbol is `SQLiteSelect`, `PgSelect`, `MySqlSelect`, etc., we extract the generic type arguments and reconstruct the row shape:

```ts
function resolveDrizzleSelect(type: ts.Type, checker: ts.TypeChecker): ts.Type {
  const symbol = type.getSymbol();
  const symbolName = symbol?.name;

  if (symbolName === "SQLiteSelect" || symbolName === "PgSelect" || symbolName === "MySqlSelect") {
    // Extract the table schema type from the first type argument
    const typeArgs = (type as ts.TypeReference).typeArguments;
    if (typeArgs && typeArgs.length > 0) {
      const tableSchema = typeArgs[0]; // { id: SQLiteInteger, email: SQLiteText, ... }
      // Walk the schema and resolve each column to its TS type
      return resolveDrizzleTableSchema(tableSchema, checker);
    }
  }

  return type;
}
```

#### Solution B: Resolve via `$inferSelect` alias

A more robust approach: instead of resolving from the query builder, we find the table's `$inferSelect` property type in the schema. But this requires the schema import to resolve, which is why using the user's `tsconfig.json` in the program is critical.

#### Solution C: Fallback to `checker.typeToString` with `NoTypeReduction`

Sometimes the TS compiler _does_ resolve the full type if you ask it the right way. Experiment with:

```ts
checker.typeToString(type, undefined, ts.TypeFormatFlags.NoTypeReduction);
```

This flag prevents the compiler from collapsing union types and may reveal the full resolved shape.

### 6.2 `$inferInsert` and `Returning` Types

For mutations:

```ts
return await ctx.db.insert(schema.users).values(args).returning();
```

The return type is `SQLiteInsert<...>` or `SQLiteInsertReturning<...>`. Same problem as `SQLiteSelect`.

**Strategy:** Build a map of Drizzle type symbol names → resolution functions:

```ts
const DRIZZLE_RESOLVERS: Record<string, (type: ts.Type, checker: ts.TypeChecker) => ts.Type> = {
  SQLiteSelect: resolveSelectReturning,
  SQLiteSelectAsync: resolveSelectReturning,
  PgSelect: resolveSelectReturning,
  MySqlSelect: resolveSelectReturning,
  SQLiteInsert: resolveInsertReturning,
  SQLiteInsertReturning: resolveInsertReturning,
  // ...
};
```

Each resolver extracts the table schema from type arguments and maps Drizzle column types (`SQLiteInteger`, `SQLiteText`, `SQLiteReal`, etc.) to TypeScript primitives (`number`, `string`, `number`).

### 6.3 Column Type Mapping

Drizzle column types carry mode information (`mode: "boolean"`, `mode: "json"`). We need to check the column's generic parameters to determine the actual TS type.

```ts
function resolveDrizzleColumnType(columnType: ts.Type, checker: ts.TypeChecker): string {
  const symbol = columnType.getSymbol();
  const name = symbol?.name;

  // Map common Drizzle SQLite column types
  if (name === "SQLiteInteger") {
    // Check if it has mode: "boolean"
    const typeArgs = (columnType as ts.TypeReference).typeArguments;
    if (typeArgs && typeArgs.length >= 2) {
      const modeType = typeArgs[1];
      if (modeType.isLiteral() && modeType.value === "boolean") {
        return "boolean";
      }
    }
    return "number";
  }

  if (name === "SQLiteText") return "string";
  if (name === "SQLiteReal") return "number";
  if (name === "SQLiteBlob") return "string | Buffer"; // or Uint8Array
  if (name === "SQLiteBoolean") return "boolean";

  // Fallback: try to resolve via the column's `$inferSelect` property
  const inferProp = checker.getPropertyOfType(columnType, "$inferSelect");
  if (inferProp) {
    const inferType = checker.getTypeOfSymbolAtLocation(inferProp, inferProp.valueDeclaration!);
    return typeToString(inferType, checker);
  }

  return "any /* unknown Drizzle column type */";
}
```

### 6.4 The Recommended Approach

Start with a **Drizzle-agnostic recursive walker** (Section 5) that works for plain types (primitives, objects, unions). Then add a **Drizzle-specific resolution layer** (Section 6) that intercepts known Drizzle query builder types and resolves them to row shapes before the generic walker sees them.

This keeps the core inliner clean and lets us incrementally add support for more Drizzle types.

## 7. Watch Mode

```ts
import chokidar from "chokidar";

export function watchAndRegenerate(entryFile: string, outFile: string) {
  const watcher = chokidar.watch(
    [
      entryFile,
      path.join(path.dirname(entryFile), "../schema.ts"),
      path.join(path.dirname(entryFile), "../types.ts"),
    ],
    { ignoreInitial: false },
  );

  let debounceTimer: ReturnType<typeof setTimeout>;

  watcher.on("change", () => {
    clearTimeout(debounceTimer);
    debounceTimer = setTimeout(() => {
      generateClientTypes(entryFile, outFile);
    }, 300);
  });

  return watcher;
}
```

**Dependency:** `chokidar` (~50KB, standard choice). Alternative: Node's `fs.watch` (brittle on Linux for recursive dirs, no cross-platform consistency).

## 8. CLI Integration

### 8.1 Command Registration

Add to `packages/cli/src/index.ts`:

```ts
import { generateClientTypesCommand } from "./commands/generateClientTypes";

cli
  .command("generate-client-types", "Generate typed client manifest from function definitions")
  .option("--watch", "Watch functions/ directory and regenerate on change")
  .option("--out <file>", "Output file path", { default: "edgepod/client.gen.ts" })
  .action(generateClientTypesCommand);
```

### 8.2 Command Implementation

```ts
// packages/cli/src/commands/generateClientTypes.ts
import path from "node:path";
import fs from "node:fs";
import { consola } from "consola";
import { findRootPath } from "../utils/findFiles";

export async function generateClientTypesCommand(options: { watch?: boolean; out?: string }) {
  const rootPath = await findRootPath();
  if (!rootPath) {
    consola.error("No package.json found. Are you in the right directory?");
    process.exit(1);
  }

  const entryFile = path.join(rootPath, "edgepod", "functions", "index.ts");
  const outFile = path.join(rootPath, options.out || "edgepod/client.gen.ts");

  if (!fs.existsSync(entryFile)) {
    consola.error(`Functions entry file not found: ${entryFile}`);
    process.exit(1);
  }

  try {
    await generateClientTypes(entryFile, outFile);
    consola.success(`Generated ${outFile}`);

    if (options.watch) {
      consola.info("Watching for changes...");
      const watcher = watchAndRegenerate(entryFile, outFile);
      process.on("SIGINT", () => {
        watcher.close();
        process.exit(0);
      });
    }
  } catch (e) {
    consola.error(`Generation failed: ${e instanceof Error ? e.message : e}`);
    process.exit(1);
  }
}
```

### 8.3 Generation Pipeline

```ts
// packages/cli/src/type-extractor/generate.ts
import { createTypeExtractor } from "./program";
import { findExportedFunctions } from "./discovery";
import { extractFunctionRouterEntry } from "./signature";
import { typeToString } from "./inliner";
import { resolveDrizzleType } from "./drizzle";
import fs from "node:fs";

export async function generateClientTypes(entryFile: string, outFile: string) {
  const { checker, sourceFile } = createTypeExtractor(entryFile);
  const exportedFunctions = findExportedFunctions(sourceFile, checker);

  const entries: Array<{
    name: string;
    args: string;
    returns: string;
  }> = [];

  for (const fn of exportedFunctions) {
    const { args, returns } = extractFunctionRouterEntry(fn.name, fn.type, checker);
    entries.push({ name: fn.name, args, returns });
  }

  const content = renderRouter(entries);
  fs.mkdirSync(path.dirname(outFile), { recursive: true });
  fs.writeFileSync(outFile, content, "utf-8");
}

function renderRouter(entries: Array<{ name: string; args: string; returns: string }>): string {
  const lines = entries.map(
    (e) => `  ${e.name}: {\n    args: ${e.args};\n    returns: ${e.returns};\n  };`,
  );

  return `// Auto-generated by EdgePod CLI. Do not edit manually.\n\nexport type EdgePodRouter = {\n${lines.join("\n")}\n};\n`;
}
```

## 11. Open Questions

1. **Should we support `export { foo } from './other'` re-exports?**
   - The current discovery logic only scans the entry file. Re-exports from other files would require following import declarations.
   - Recommendation: Start without re-export support. Document that all functions must be defined in `edgepod/functions/index.ts`.

2. **What about overloaded functions?**
   - If a user defines `function foo(ctx)` and `function foo(ctx, args)`, we take the first call signature.
   - Recommendation: Warn if multiple signatures detected, use the first.

3. **How to handle `any` and `unknown` in generated types?**
   - If a function returns `Promise<any>`, we emit `returns: any`.
   - This is honest and correct. The user should add explicit return type annotations.

4. **Should we include JSDoc comments in generated types?**
   - Not for v1.0. The generated file is purely for type inference.

5. **Monorepo shortcut?**
   - For users with frontend and backend in the same repo, `import type * as Functions` works today (but leaks Drizzle types into frontend compilation).
   - The generated file is still recommended even in monorepos for clean separation.

## Appendix A: Expected Generated Output for Template Functions

Given the template `functions.ts`:

```ts
export const getUsers = async (ctx: Ctx) => {
  return await ctx.db.select().from(schema.users);
};

export const getUserById = async (ctx: Ctx, args: { id: number }) => {
  const result = await ctx.db
    .select()
    .from(schema.users)
    .where(eq(schema.users.id, args.id))
    .limit(1);
  return result[0] || null;
};

export const createUser = async (ctx: Ctx, args: { email: string; name: string }) => {
  const newUser = await ctx.db.insert(schema.users).values(args).returning();
  return newUser[0];
};
```

With a schema like:

```ts
export const users = sqliteTable("users", {
  id: integer("id", { mode: "number" }).primaryKey({ autoIncrement: true }),
  email: text("email").notNull(),
  name: text("name"),
});
```

The generator should produce:

```ts
// Auto-generated by EdgePod CLI. Do not edit manually.

export type EdgePodRouter = {
  getUsers: {
    args: never;
    returns: Array<{ id: number; email: string; name: string | null }>;
  };
  getUserById: {
    args: { id: number };
    returns: { id: number; email: string; name: string | null } | null;
  };
  createUser: {
    args: { email: string; name: string };
    returns: { id: number; email: string; name: string | null };
  };
};
```

This is the **north star** — the implementation should converge on this output.
